Project overview

API2File

A native macOS app and CLI that turns remote APIs into local working files, then keeps both sides in sync. Think Dropbox for APIs: cloud data becomes CSV, Markdown, HTML, JSON, ICS, VCF, and more.

Bidirectional Pull from APIs, edit locally, push back automatically.
Config-driven New services are described in adapter JSON, not custom Swift code.
Agent-ready Every sync folder gets a generated CLAUDE.md guide.
15 formats CSV, JSON, HTML, Markdown, YAML, ICS, VCF, EML, SVG, XLSX, DOCX, PPTX, text, raw, media.
12 adapters Bundled demos plus cloud services like Monday.com, Wix, GitHub, and Airtable.
Menu bar + CLI Always-on sync from a native app, or scriptable control from the terminal.
Git-backed Every service folder is its own repo with automatic sync commits and rollback history.
Slide 1
What it does

It converts APIs into normal files people and tools already know how to use.

Instead of forcing users into a web dashboard for every system, API2File materializes remote resources as local files inside a sync folder. A task board can look like a spreadsheet, contacts become .vcf, calendar items become .ics, pages become .html, and service records can be JSON files.

For humans

Use Numbers, Excel, Calendar, Contacts, Preview, Keynote, VS Code, or any editor instead of learning yet another admin UI.

For operations

Keep a clean local copy of service data, diff it with git, search it instantly, and automate around standard files.

For developers

Connect new services through declarative adapter configs that define auth, endpoints, transforms, file mappings, and sync timing.

For AI workflows

Agents can read the generated guides, inspect the file tree, edit content safely, and let API2File handle the API calls.

Simple setup path

$ swift run api2file init
$ swift run api2file add demo
$ swift run api2file sync demo

~/API2File-Data/
  CLAUDE.md
  demo/
    CLAUDE.md
    tasks.csv
    incidents.csv
    pages/home.html
    events/team-standup.ics

Core promise

User files stay clean. Sync metadata is hidden in .api2file/state.json, while the visible files contain just the real content.

This is not just “export data to files.” It is a durable working model where files are the user interface and APIs are the transport layer.

Slide 3
Main flow 1

Pull: API to local files

The pull path fetches remote data, transforms it, maps it to files, writes to disk, updates sync state, and commits the result to git.

Fetch

HTTPClient calls REST or GraphQL endpoints with the adapter’s auth, headers, pagination, and retry rules.

→

Extract

JSONPath selects the records from the response, such as $.data.items[*].

→

Transform

Declarative pull transforms can pick, omit, rename, flatten, or reshape the data.

→

Write files

FileMapper decides paths and FormatConverter writes CSV, Markdown, HTML, JSON, ICS, and more.

→

Record state

State is stored in .api2file/state.json and the folder gets an automatic pull commit.

User-facing result

A fresh local workspace appears with normal files that can be opened by Finder, editors, desktop apps, scripts, and agents.

Slide 4
Main flow 2

Push: local edits back to the API

When a user or agent saves a file, API2File watches the change, decodes the file back into records, computes the diff, and issues the right API mutations.

Detect file save

FSEvents sees the change and the coordinator debounces quick saves into one push.

→

Decode content

The format converter turns local CSV, JSON, ICS, VCF, Markdown, and others back into record data.

→

Diff records

Sync state tells the engine what is new, changed, or deleted so the API calls can be precise.

→

Call the API

Configured create, update, and delete endpoints are invoked with transformed payloads.

→

Refresh and commit

State updates, user files can be regenerated from canonical pull transforms, and git captures the push.

Important UX detail

Users edit content, not transport details. IDs stay in stable columns or fields, while the engine handles API semantics under the hood.

Slide 6
Main flow 4

Using the files with Claude Code and other coding agents

API2File generates a CLAUDE.md guide inside the root sync folder and inside each service folder. That gives agents the local operating manual: what resources exist, which file formats they map to, how to create or update records, sync timing, and constraints such as “do not modify _id.”

Agent sees structure

The guide explains folders like contacts/, events/, notes/, and which files are collections versus one-per-record resources.

Agent edits normal files

Claude Code can update Markdown notes, append rows to CSV, change JSON fields, or create new record files without learning the remote API schema first.

API2File does the transport

The file watcher turns those saves into create, update, and delete API calls, then refreshes sync state and git history.

Human and AI stay aligned

Both work against the same visible local files, so reviews, diffs, and collaboration happen in one shared workspace.

What the generated guide tells an agent

Demo API - Synced via API2File

To update: Edit the file and save.
To create: Add a new row or create a new file.
To delete: Delete the file.

Sync behavior:
- Poll interval: 10 seconds
- Local changes push within 500ms
- All changes are git-committed automatically

Constraints:
- Do not edit .api2file/
- Do not modify _id fields

Why this matters

Instead of building custom tool integrations per service, you can let the agent operate on a file system contract that is already friendly to coding tools.

Slide 7
Product surfaces

How users interact with the product

API2File is intentionally flexible: menu bar app for always-on desktop usage, CLI for setup and scripts, adapter configs for extending services, and a local control API for automation.

Menu bar app

Native SwiftUI menu bar experience with service discovery, status, sync-now actions, pause, and preferences.

CLI

init, add, sync, pull, status, and list support terminal-based workflows.

Adapter configs

JSON config defines auth, endpoints, file mapping, transforms, and sync rules. New services usually mean new config, not engine code.

Control API

Local endpoints such as localhost:21567/api/services/demo/sync can trigger syncs or expose status to other tools.

Typical commands

$ swift run api2file init
$ swift run api2file add demo
$ swift run api2file status
$ swift run api2file pull demo
$ swift run api2file sync demo
$ swift run API2FileApp

Architecture note

One engine, many services. The top-level sync engine orchestrates multiple adapter engines, one per connected service.
